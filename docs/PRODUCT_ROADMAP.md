# MailTalker 产品目标文档

## 愿景

构建一个**零基础设施依赖**的端对端加密通信工具——利用全球已有的邮件基础设施作为消息传输层，让任何拥有邮箱的人都能进行隐秘、安全的实时对话。

## 产品路线图

### Phase 2: 安全性与可靠性 (v2.0)

#### 2.1 密钥交换协议
**问题**: 当前需要用户手动线下传递 64 位 hex 密钥，体验差且容易出错。
**方案**: 实现 ECDH (Elliptic Curve Diffie-Hellman) 密钥交换。
- 用户 A 生成临时密钥对，将公钥通过邮件发送给 B
- 用户 B 生成自己的密钥对，用 A 的公钥 + 自己的私钥派生共享密钥
- B 将自己的公钥回传给 A
- 双方独立计算出相同的 AES-256 密钥
- 加入 SAS (Short Authentication String) 验证防止中间人攻击

```
技术栈: Node.js crypto.ECDH (curve25519 或 secp256k1)
```

#### 2.2 消息签名与防篡改
- 每条消息附加 HMAC-SHA256 签名
- 接收方验证签名，拒绝篡改消息
- 消息序号防重放攻击

#### 2.3 前向保密 (Forward Secrecy)
- 实现 Double Ratchet 协议（Signal 协议的核心）
- 每条消息使用不同的会话密钥
- 即使长期密钥泄露，历史消息仍然安全

---

### Phase 3: 隐蔽性增强 (v3.0)

#### 3.1 可配置邮件伪装
**问题**: 固定主题 `[*MT-SecMSG*]` 太明显。
**方案**:
- 允许双方约定自定义主题行模板（如 "Re: Meeting Notes"）
- 支持生成逼真的邮件正文（看起来像正常商务邮件）
- 载荷 PNG 的文件名随机化（不再固定 payload.png）

#### 3.2 真正的图片隐写术 (Steganography)
**问题**: 当前 PNG 是明显的噪声图，一看就不正常。
**方案**:
- 将加密数据嵌入到真实照片的 LSB (Least Significant Bit) 中
- 用户可选择载体图片（风景照、食物照等）
- 修改后的图片肉眼看不出区别
- 使用 DCT 域隐写增强抗检测能力

```
技术栈: sharp (高性能图片处理) 替代 jimp
```

#### 3.3 流量模式混淆
- 定时发送空/假消息，掩盖真实通信频率
- 消息长度填充到固定大小，防止流量分析
- 可配置"静默时段"

---

### Phase 4: 用户体验 (v4.0)

#### 4.1 多媒体消息
- 支持发送图片、文件、语音消息
- 大文件自动分片（受邮件附件大小限制，通常 25MB）
- 文件加密后分块编码为多张 PNG

#### 4.2 群组聊天
- 基于共享群密钥的多人聊天
- 群密钥轮换机制（成员变动时）
- 群消息使用多播（发送到每个成员的邮箱）

#### 4.3 消息管理
- 消息搜索（本地全文检索）
- 阅后即焚（定时自动删除）
- 消息引用/回复
- 消息状态（已发送/已送达，基于 IMAP 回执）

#### 4.4 跨平台
- macOS 和 Linux 打包分发
- 移动端 (React Native / Capacitor)
- 数据库跨设备同步（通过专用同步邮件频道）

---

### Phase 5: 去中心化增强 (v5.0)

#### 5.1 多邮箱路由
- 支持配置多个邮箱账户
- 消息通过不同邮箱轮流发送，降低单点封禁风险
- 自动切换故障邮箱

#### 5.2 中继节点
- 可选的多跳路由：A → 中继邮箱 → B
- 隐藏真实发件人地址
- 中继节点看不到消息内容（端对端加密）

#### 5.3 备用传输层
- 插件化传输层架构
- 除邮件外支持: Matrix 协议、Nostr、XMPP
- 传输层故障自动切换

## 技术栈规划

### 当前 (v1.x)
| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | Electron 40 | 桌面跨平台 |
| 前端 | React 19 + Tailwind CSS 4 | UI 渲染 |
| 构建 | Webpack 5 | 模块打包 |
| 数据库 | better-sqlite3 | 本地持久化 |
| 加密 | Node.js crypto (AES-256-GCM) | 消息加密 |
| 图片 | Jimp | PNG 编解码 |
| 邮件 | nodemailer + imap-simple | SMTP/IMAP |
| 打包 | electron-builder | 分发构建 |

### 建议升级 (v2.x+)
| 层级 | 技术 | 升级理由 |
|------|------|---------|
| 前端 | React 19 + Tailwind CSS 4 | 保持不变，足够好 |
| 构建 | **Vite** 替代 Webpack | 开发热更新快 10x，配置更简洁 |
| 图片 | **sharp** 替代 Jimp | 性能高 10-50x，支持真正隐写术 |
| 加密 | **libsodium (sodium-native)** | 更安全的密码学原语，X25519 + XChaCha20-Poly1305 |
| 状态 | **Zustand** | 替代 prop drilling，轻量全局状态 |
| ORM | **Drizzle ORM** (可选) | 类型安全的 SQL 查询 |
| 测试 | **Vitest** 替代 Jest | 与 Vite 集成，速度更快 |
| 邮件 | **imapflow** 替代 imap-simple | 更现代，支持 IDLE 推送（无需轮询） |

### 关键技术决策

**为什么保留 Electron 而不换 Tauri?**
- better-sqlite3 和 nodemailer 是 Node.js 原生库，迁移 Tauri (Rust) 成本极高
- Electron 生态成熟，调试工具完善
- 体积可通过 electron-builder 的 asar 压缩优化

**为什么推荐 imapflow?**
- 支持 IMAP IDLE 命令：服务器主动推送新邮件通知
- 不再需要 10 秒轮询，延迟从 ~10s 降至 ~1s
- 内存占用更低，连接更稳定

**为什么推荐 libsodium?**
- 自带 X25519 ECDH 密钥交换（Phase 2 必需）
- XChaCha20-Poly1305 比 AES-GCM 更适合软件实现（无需 AES-NI 硬件加速）
- 常量时间操作，防侧信道攻击
- API 设计更难误用

## 设计原则

1. **邮件是基础设施，不是产品** — 用户永远不应该感觉在"发邮件"，体验应该像 Signal/Telegram
2. **零信任** — 假设邮件服务商是敌对的，只信任本地加密
3. **渐进式安全** — 默认安全，高级用户可配置更强的保护
4. **离线优先** — 所有数据本地存储，无网络也能浏览历史消息
5. **最小权限** — 渲染进程沙箱化，只通过 preload API 访问系统资源

## 竞品对比

| 特性 | MailTalker | Signal | Telegram | ProtonMail |
|------|-----------|--------|----------|------------|
| 无需注册/手机号 | ✅ | ❌ | ❌ | ❌ |
| 无中心服务器 | ✅ | ❌ | ❌ | ❌ |
| 利用现有基础设施 | ✅ (邮件) | ❌ | ❌ | ✅ (邮件) |
| E2E 加密 | ✅ | ✅ | 部分 | ✅ |
| 隐蔽性 | ✅ (伪装邮件) | ❌ | ❌ | ❌ |
| 抗封锁 | ✅ (邮件难封) | 中 | 低 | 中 |
| 实时性 | 低 (~10s) | 高 | 高 | 低 |

**核心差异化**: MailTalker 是唯一一个不需要专用服务器、不需要注册账号、利用全球邮件基础设施实现加密通信的工具。在网络审查严格的环境中，封锁邮件协议的代价远高于封锁 Signal/Telegram 的服务器。
